# 第 02 章：代码库全景图

这一章的目标很简单：

> 在正式钻进训练代码之前，先建立一张“仓库地图”。

如果没有这张地图，你后面会遇到两个常见问题：

- 看见 `verl/` 这种大目录，不知道哪些是 Search-R1 真正新增的逻辑；
- 看见 `search_r1/` 里的检索代码，又不知道它和训练主循环是怎么接上的。

所以这一章不追求细，而是追求**方向感**。

---

## 1. 顶层目录先看什么

在仓库根目录下，你最应该优先关注这些内容：

| 路径 | 作用 | 现在是否建议细读 |
| --- | --- | --- |
| `README.md` | 项目介绍、安装、快速开始、训练/推理说明 | 是 |
| `infer.py` | 最小推理闭环示例 | 是 |
| `train_ppo.sh` | PPO 训练启动脚本 | 是 |
| `train_grpo.sh` | GRPO 训练启动脚本 | 是 |
| `retrieval_launch.sh` | 本地检索服务启动方式 | 是 |
| `search_r1/` | Search-R1 自己的核心逻辑 | 是 |
| `verl/` | 底层 RL 训练框架 | 是，但先抓主线 |
| `docs/` | 检索器、多机训练等补充文档 | 可以按需看 |
| `example/` | 示例语料与样例文件 | 建议顺手看 |
| `scripts/` | 数据下载、上传、数据处理辅助脚本 | 暂时后看 |

如果让我用一句话概括：

- `search_r1/` 是“Search-R1 特有逻辑”；
- `verl/` 是“承载 RL 训练的底座”。

## 2. 先分清：这个仓库不是从零写的

README 一开始就明确写了：

> Search-R1 built upon veRL.

这句话非常重要，因为它解释了为什么仓库里会有一个体量很大的 `verl/` 目录。

你可以把它理解成：

- `verl/` 提供通用 RL 训练能力；
- Search-R1 在这个基础上增加了“搜索型 agent rollout”。

所以你在读代码时，要不断问自己一个问题：

> 这是通用 RL 框架代码，还是 Search-R1 为“搜索工具调用”新增的代码？

这个区分会极大提高阅读效率。

---

## 3. 从架构上看，仓库可以分成三层

我建议你把整个项目拆成三层看。

### 3.1 第一层：应用层（Search-R1 自己新增的核心）

最关键的目录是：

- `search_r1/llm_agent/`
- `search_r1/search/`

这一层回答的问题是：

- 模型如何触发搜索；
- 搜索结果如何回到上下文；
- 检索服务如何提供文档；
- 多轮推理/搜索如何组织。

这里最关键的文件包括：

- `search_r1/llm_agent/generation.py`
- `search_r1/search/retrieval_server.py`
- `search_r1/search/retrieval.py`

### 3.2 第二层：训练调度层（veRL trainer / worker）

核心目录是：

- `verl/trainer/`
- `verl/workers/`

这一层回答的问题是：

- 训练入口如何启动；
- actor、critic、reference policy、reward model 如何协同；
- rollout 数据怎样流过训练循环；
- 分布式训练如何调度。

这里最关键的文件包括：

- `verl/trainer/main_ppo.py`
- `verl/trainer/ppo/ray_trainer.py`
- `verl/workers/actor/*`
- `verl/workers/critic/*`
- `verl/workers/rollout/*`

### 3.3 第三层：算法层（reward / advantage / loss）

核心文件是：

- `verl/trainer/ppo/core_algos.py`
- `verl/utils/reward_score/qa_em.py`
- `verl/utils/dataset/rl_dataset.py`

这一层回答的问题是：

- reward 怎么算；
- advantage / return 怎么算；
- PPO / GRPO 的差别在哪里；
- 数据集样本长什么样。

## 4. 如果用一句话说每个关键路径的职责

### 4.1 `infer.py`

这是**最小可运行示例**。

它不负责训练，只负责演示：

- 模型怎么输出 `<search>query</search>`；
- 系统怎么调用搜索引擎；
- 怎么把结果回填给模型继续生成。

你可以把它当成：

> “Search-R1 思想的教学版 demo”。

### 4.2 `search_r1/llm_agent/generation.py`

这是 Search-R1 最关键的应用层文件。

它负责把普通的“一次性文本生成”改造成：

```text
生成 -> 解析动作 -> 调搜索 -> 拼 observation -> 再生成
```

这一层是 Search-R1 真正区别于普通 RLHF/PPO 的地方。

### 4.3 `search_r1/search/retrieval_server.py`

这是一个 FastAPI 服务，暴露 `/retrieve` 接口。

它的作用很纯粹：

- 收到若干查询；
- 调 retriever；
- 返回 top-k 文档。

所以 Search-R1 的“搜索工具”，本质上只是一个独立服务。

### 4.4 `verl/trainer/main_ppo.py`

这是训练总入口。

它负责：

- 解析配置；
- 初始化 tokenizer；
- 建 worker；
- 创建 reward manager；
- 启动 `RayPPOTrainer`。

### 4.5 `verl/trainer/ppo/ray_trainer.py`

这是训练主循环。

如果你以后只能深读一个大文件，我最推荐这个。

因为它集中串起了：

- dataloader
- rollout
- reward
- KL
- advantage
- actor / critic update
- validation
- checkpoint

### 4.6 `verl/trainer/ppo/core_algos.py`

这是算法核心。

它不关心搜索服务，也不关心具体任务，只关心：

- GAE 怎么算；
- GRPO outcome advantage 怎么算；
- PPO loss / value loss 怎么算；
- KL penalty 怎么算。

## 7. 一个很重要的阅读策略：不要一开始就想“全懂”

这个仓库有不少工程性代码，尤其是 `verl/` 部分。

如果你一开始就试图逐行看懂所有函数，很容易出现两种情况：

- 被大量分布式和 batch 操作细节干扰；
- 看到了实现，但没抓住作者真正想解决的问题。

所以更好的策略是分三轮：

### 第一轮：只看“主链路”

只回答：
- 训练从哪里启动；
- rollout 在哪里发生；
- 搜索在哪里接入；
- reward 在哪里计算；
- update 在哪里执行。

### 第二轮：看“关键接口”

重点看：
- `run_llm_loop()`
- `retrieve_endpoint()`
- `RewardManager.__call__()`
- `compute_advantage()`

### 第三轮：再看张量细节

例如：
- padding / masking
- info_mask
- batch union / pop / repeat
- reference policy KL
- critic value

这时你的理解会稳很多。
