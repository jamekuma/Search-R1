# 第 10 章：检索服务、语料与索引

讲到这里，你已经知道 Search-R1 的核心主线是：

- LLM 在生成过程中输出 `<search>`；
- 系统把 query 发给检索服务；
- 检索服务返回文档；
- 文档被包装成 `<information>` 再喂回模型。

但如果不理解搜索系统本身，你对 Search-R1 的理解还是会缺一块。

这一章就专门回答：

> Search-R1 的“搜索引擎”到底是什么？它支持哪些形式？本地语料、索引和检索服务之间是什么关系？

---

## 1. 先分清三个概念

很多人第一次接触 Search-R1，会把下面三个概念混在一起：

1. **语料（corpus）**
2. **索引（index）**
3. **检索服务（retrieval server）**

其实它们分别扮演不同角色。

### 1.1 语料（corpus）

这是原始文档集合。

在 Search-R1 里，推荐格式是 jsonl，每行至少包含：
- `id`
- `contents`

参考：
- `README.md:183-194`
- `example/corpus.jsonl:1-5`

### 1.2 索引（index）

这是为了高效检索而提前构建的结构。

例如：
- BM25 的 Lucene 索引；
- Dense Retriever 的 FAISS 索引；
- ANN 的 HNSW 索引。

### 1.3 检索服务（server）

这是一个 HTTP 层包装器。它接收 query，调用底层 retriever，再返回文档。

所以三者关系是：

```text
corpus -> build index -> retriever -> retrieval server -> Search-R1
```

---

## 2. Search-R1 的一个重要设计哲学

README 里有一句话非常关键：

> The main philosophy is to launch a local or remote search engine server separately from the main RL training pipeline.

也就是：

> 检索服务和 RL 训练主流程是解耦的。

顺手提醒一下：README 在 `README.md:211` 处写的是 `search_r1/search/retriever_server.py`，但当前仓库真实文件名其实是 `search_r1/search/retrieval_server.py`。如果你按 README 的名字搜不到文件，不是你看错，而是 README 这里本身有个命名笔误。

这意味着训练代码只依赖一个统一接口：

- `http://127.0.0.1:8000/retrieve`

而不强依赖某个具体检索器实现。

所以你后面如果想换成：
- BM25
- Dense Retriever
- ANN
- Google Search API
- SerpAPI
- 你自己的领域搜索引擎

从框架角度都比较容易。

---

## 3. Search-R1 支持哪些检索器形态

根据：
- `docs/retriever.md:1-127`
- `README.md:203-211`

大致支持三类：

### 3.1 本地 sparse retriever
例如：
- BM25

优点：
- 快；
- 不需要 GPU；
- 工程简单。

缺点：
- 在某些语义匹配要求高的场景里，效果不如 dense retriever。

### 3.2 本地 dense retriever
例如：
- e5
- bge 等 embedding 检索器

优点：
- 语义检索能力更强；
- 更适合复杂问答场景。

缺点：
- 需要 embedding 模型；
- 通常依赖 FAISS；
- 资源开销更高。

### 3.3 在线搜索引擎
例如：
- Google Search API
- SerpAPI

优点：
- 获取最新知识；
- 可覆盖开放域网页信息。

缺点：
- 花钱；
- 网络稳定性和配额会影响训练；
- 可复现性更差。

## 4. `retrieval_server.py`：Search-R1 看到的“搜索引擎”长什么样

文件：
- `search_r1/search/retrieval_server.py:324-392`

如果用一句话概括它的职责：

> 把本地 retriever 封装成统一的 FastAPI 服务。

### 4.1 输入格式

服务暴露的是：

- `POST /retrieve`

请求体大致是：

```json
{
  "queries": ["What is Python?"],
  "topk": 3,
  "return_scores": true
}
```

### 4.2 输出格式

返回大概是：

```json
{
  "result": [
    [
      {"document": {...}, "score": ...},
      ...
    ]
  ]
}
```

### 4.3 为什么这个接口设计很关键

因为 Search-R1 的训练和推理代码只需要知道：

- 我把 query 列表发过去；
- 我能拿回一批文档。

它不需要关心底层到底是 BM25、FAISS 还是在线 API。

这就是“解耦”的真正体现。

---

## 5. `retrieval.py`：真正的检索器逻辑在哪里

文件：
- `search_r1/search/retrieval.py:132-323`

这里定义了两个最核心的检索器类：

- `BM25Retriever`
- `DenseRetriever`

以及一个统一工厂：

- `get_retriever(config)`

### 5.1 `BM25Retriever`

位置：
- `retrieval.py:165-224`

它本质上是对 Pyserini / Lucene 索引的封装。

你可以把它理解成：

- 用关键词匹配查 top-k；
- 如果 index 里自带文档内容，就直接取；
- 否则再从 corpus 文件里按 docid 回查。

### 5.2 `DenseRetriever`

位置：
- `retrieval.py:236-309`

它的基本流程是：

1. 把 query 编码成 embedding；
2. 用 FAISS 在向量空间里搜索 top-k；
3. 根据返回的 idx 去 corpus 中取文档。

你会发现，这其实就是很典型的 dense retrieval 流程。

---

## 6. Dense Retriever 的工作流，和你熟悉的深度学习有什么关系

因为你的背景是深度学习和 PyTorch，所以这里我用更贴近你直觉的方式讲。

Dense retriever 可以理解成：

- 先用 encoder 把 query 编成向量；
- 把文档也提前离线编码成向量；
- 用内积 / 相似度做最近邻检索。

在 Search-R1 的代码里：

### 6.1 query 编码

- `retrieval.py:73-129`

`Encoder.encode()` 会：
- tokenize query；
- 送入 retrieval model；
- 做 pooling；
- 得到 float32 embedding。

### 6.2 向量检索

- `retrieval.py:260-309`

调用的是：

```python
batch_scores, batch_idxs = self.index.search(batch_emb, k=num)
```

这一步你可以直接理解成：

- 用 FAISS 做 top-k nearest neighbor。

### 6.3 再映射回原文档

之后会根据 `batch_idxs` 去 `corpus` 里取回对应文档。

所以整体流程和你熟悉的“embedding 检索”几乎完全一致。

---

## 7. corpus 的格式为什么设计成 `contents`

README 和示例文件都建议：

```json
{"id": "0", "contents": "\"Title\"\nBody"}
```

这样设计有两个明显好处：

### 7.1 对 BM25 和 Dense 都统一

无论是关键词检索还是向量检索，底层都可以直接把 `contents` 当作统一文档文本。

### 7.2 回到 LLM 侧更容易格式化

你会看到 `generation.py` 和 `infer.py` 在把文档拼给模型时，都会做类似处理：

- 第一行当 title
- 后面当正文

所以这种格式既方便索引，也方便回填给 LLM。

## 8. 如果你想自己建索引，代码在哪里

核心文件：
- `search_r1/search/index_builder.py:297-349`
- `search_r1/search/build_index.sh:2-19`

你可以把索引构建看成两类。

### 8.1 BM25 索引

调用 Pyserini 建 Lucene 索引。

相关逻辑在：
- `index_builder.py:129-162`

### 8.2 Dense / ANN 索引

先离线编码文档，再用 FAISS 建索引。

相关逻辑在：
- `index_builder.py:245-286`

支持：
- `Flat`
- `HNSW32/64/128`
- 等不同 faiss 类型

所以 Search-R1 不是只能用作者给的索引，而是从工程上支持你替换自己的语料与检索器。

## 9. 为什么 `docs/retriever.md` 值得认真读

这个文档不是泛泛而谈，而是给了很实用的选择建议：

- `docs/retriever.md:7-25`

比如：

### 9.1 私有或垂直领域语料
优先本地 retriever。

### 9.2 领域内没有强 embedding retriever
优先 sparse（如 BM25）。

### 9.3 有 GPU，追求高质量
优先 flat dense indexing。

### 9.4 GPU 不够，追求工程可跑
优先 ANN。

### 9.5 想做开放域、最新知识搜索
优先在线搜索引擎。

这其实已经是在帮你做“系统设计选型”。

## 10. Search-R1 为什么能比较容易换搜索引擎

现在你应该能总结出原因了：

### 原因一：统一的 HTTP 接口
trainer 和 infer 只认：
- `/retrieve`
- `queries/topk/return_scores`
- 返回文档列表

### 原因二：统一的文档格式
底层 retriever 只要最后能返回带 `contents` 的文档，就能接上。

### 原因三：搜索逻辑被隔离在 `search_r1/search/`
LLM rollout 不需要感知内部实现。

这也是为什么 Search-R1 的系统设计很适合做二次改造。

## 12. 本章小结

这一章最关键的结论有六个：

1. **Search-R1 的搜索系统和 RL 训练主流程是解耦的。**
2. **搜索系统可以拆成 corpus、index、retriever、server 四层。**
3. **Search-R1 支持 sparse、dense、ANN 和在线搜索引擎。**
4. **训练和推理代码只依赖统一的 `/retrieve` HTTP 接口。**
5. **`retrieval.py` 是底层检索逻辑核心，`retrieval_server.py` 是服务封装层。**
6. **这种设计让 Search-R1 很适合替换成你自己的语料和搜索引擎。**

下一章，我们就从“理解原项目”走到“如何改造和复用”——如果你想把 Search-R1 迁移到自己的任务，应该改哪里，不应该乱改哪里。
